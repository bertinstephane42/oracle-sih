Module 1 - Multitenant CDB/PDB
Objectif de la Simulation

Ce module modelise l'architecture Multitenant d'Oracle 12c+ dans un contexte CHU. Il demontre trois phenomenes internes critiques :

  • Isolation logique CDB/PDB : Le CDB$ROOT heberge le dictionnaire de donnees partage (SYSTEM, SYSAUX, Undo). Chaque PDB (Pluggable Database) possede ses propres tablespaces utilisateurs mais partage les structures de controle. L'isolation est logique, non physique : une contention sur le dictionnaire partage affecte tous les PDBs.
  • Resource Manager : Le DEFAULT_CDB_PLAN distribue les ressources CPU/IO via des shares proportionnelles. Les PDBs critiques (Urgences, Bloc operatoire) recoivent des shares eleves garantissant un SLA meme sous contention. Les PDBs de recherche fonctionnent en best-effort.
  • Data Masking RGPD : Lors du clonage PDB (production vers test/recherche), DBMS_DATA_MASKING anonymise les champs sensibles (NIR, nom, diagnostic) pour conformite RGPD Art. 25 (Privacy by Design).
Controles Disponibles
  • Sliders de shares : Ajustez les parts CPU/IO pour chaque PDB (0 a 10). Le pourcentage alloue est calcule proportionnellement au total des shares. Un PDB avec 0 share ne recoit aucune ressource garantie.
  • Appliquer le Plan : Valide la repartition et affiche un diagnostic de conformite SLA (les PDBs critiques doivent avoir >= 50% du total).
  • Reset Defaut : Restaure les shares par defaut (Urgences:4, Bloc:3, PMSI:2, Recherche:1).
  • Simuler Pic de Charge : Genere une sequence temporelle simulant la contention sur les latches row cache objects et shared pool quand 4 PDBs accedent simultanement au dictionnaire partage du CDB$ROOT.
  • Executer DBMS_DATA_MASKING : Applique le masque sur les champs NIR et nom dans le PDB cible, demontrant le processus de clonage conforme RGPD.
Indicateurs Cles
  • Latch Waits/sec : Nombre d'attentes sur les latches du dictionnaire. Un pic indique une contention qui degrade le temps de reponse de tous les PDBs.
  • Dict Cache Hit Ratio : Taux de hit du dictionnaire de donnees en memoire. En dessous de 95%, les hard parses augmentent, degradant les performances.
  • CPU % par PDB : Repartition proportionnelle issue des shares. Verifiez que les PDBs critiques (Urgences, Bloc) totalisent au moins 50%.
Contexte HDS / CHU

Dans un CHU, l'isolation multitenant permet de separer les environnements (production, recherche, test) sur une seule instance Oracle, reduisant les couts de licence et d'administration. Le Data Masking est obligatoire pour tout clonage vers un environnement non-production contenant des donnees de santé. Les Resource Plans garantissent que les processus critiques (Urgences, Bloc operatoire) ne sont jamais affames par des requetes de recherche ou des batchs PMSI/T2A.

Module 2 - SGA / PGA Internals
Objectif de la Simulation

Ce module modelise la gestion memoire interne d'Oracle, couvrant trois mecanismes algorithmiques fondamentaux :

  • Buffer Cache LRU / Touch Count : Oracle utilise un algorithme de remplacement de blocs base sur le Touch Count (introduit en 10g). Les blocs accedes frequemment migrent vers la "hot end" de la liste LRU. Les full table scans inserent les blocs au "midpoint" avec TC=0, evitant de polluer le cache avec des donnees a usage unique.
  • Latch Cache Buffers Chains (CBC) : Les latches protegent les chaines de hash du buffer cache. Quand N sessions accedent concurrentiellement au meme block (hot block), la contention est quadratique : le temps total d'attente croit comme O(N^2). Le mecanisme de spin/sleep determine la gravite de la contention.
  • PGA Work Areas & Temp Spill : Chaque session dispose d'une zone memoire privee (PGA) pour les operations de tri, hash join, et bitmap merge. Quand l'operation excede la memoire allouee (PGA_AGGREGATE_TARGET), Oracle bascule sur des Temp Segments en disque, multipliant la latence par 500 a 10000.
Controles Disponibles
  • SGA_TARGET slider : Redimensionne la taille totale de la SGA (1024 a 16384 MB). Le Auto-Tune (AMM) analyse les advisors internes et propose une valeur optimale.
  • Acces Bloc (Touch) : Incremente le Touch Count d'un bloc aleatoire, simulant un acces recurrent. Observez la migration vers la HOT END.
  • Full Table Scan : Simule un parcours sequentiel complet : les blocs touches recoivent TC=0 et restent a la COLD END grace au Midpoint Insertion.
  • Checkpoint / Flush : Declenche un checkpoint : tous les dirty blocks sont marques pour ecriture par DBWR, les TC sont decrementes.
  • Sessions Concurrentes (latch) : Definit le nombre de sessions accedant au meme hot block (1 a 32). La contention croit de maniere quadratique.
  • Taille du Tri / PGA Limit : Definit la taille de l'operation de sort et le plafond PGA. Si le tri excede 20% du PGA (work area), un Temp Spill se produit.
Indicateurs Cles
  • Touch Count (TC) : Blocs avec TC >= 3 migrent vers la HOT END. TC = 0 signifie bloc froid (FTS ou jamais accede).
  • DIRTY / PINNED : Un bloc DIRTY a ete modifie en memoire mais pas encore ecrit sur disque. Un bloc PINNED est en cours d'utilisation par une session.
  • Wait Time Total (latch) : Temps cumule d'attente sur le latch CBC. Comparez les valeurs entre 4 et 32 sessions pour observer la croissance quadratique.
  • Mode execution (PGA) : OPTIMAL (tout en memoire) vs SPILLED (bascule disque). Le temps total en mode SPILLED inclut la penalite I/O disque (50ms/MB vs 0.1ms/MB en memoire).
Contexte HDS / CHU

Les requetes PMSI/T2A de facturation impliquent des aggregations sur des millions de sejours (RSA). Un Temp Spill sur ces requetes peut bloquer la production T2A pendant plusieurs minutes, impactant la facturation mensuelle du CHU. La contention sur les latches CBC peut survenir quand plusieurs sessions accedent aux memes blocs de reference (tables de parametres, nomenclatures). Le dimensionnement correct du SGA (via AMM) et du PGA garantit des temps de reponse stables pour les workflows critiques.

Module 3 - I/O & Coherence
Objectif de la Simulation

Ce module modelise le cycle de vie complet d'une transaction Oracle et les mecanismes de garantie de coherence, couvrant quatre concepts fondamentaux :

  • Cycle Redo/Undo : Chaque DML genere deux traces : les Undo blocks (avant-image, pour le rollback et la read consistency) dans le tablespace UNDO, et les Redo entries (apres-image, pour la recovery) dans le Redo Log Buffer. Au commit, LGWR force l'ecriture synchrone des redo entries sur disque. Les dirty blocks sont ecrits asynchronement par DBWR (lazy write).
  • SCN (System Change Number) : Horloge logique d'Oracle, incrementee a chaque commit. Le SCN garantit la read consistency : une requete voit les donnees telles qu'elles etaient a son SCN de depart, reconstruites via l'Undo si necessaire.
  • Instance Recovery : Apres un crash (SHUTDOWN ABORT), SMON execute deux phases : Redo Apply (roll-forward, rejoue tous les redo records depuis le dernier checkpoint) puis Undo Apply (rollback des transactions non commitrees). La base est alors ouverte et coherente.
  • ORA-01555 Snapshot Too Old : Erreur critique survenant quand les donnees Undo necessaires a la read consistency d'une longue requete ont ete ecrasees par des DMLs concurrents. Le risque depend de UNDO_RETENTION, de la duree de la requete, et du taux de rotation Undo.
Controles Disponibles
  • Executer Transaction Complete : Anime le cycle BEGIN -> DML (Undo) -> Redo -> COMMIT (LGWR) -> DBWR, avec changement de couleur des marqueurs a chaque etape.
  • Simuler Crash (SHUTDOWN ABORT) : Simule un crash instance : les dirty blocks en Buffer Cache sont perdus. Les marqueurs passent en rouge (erreur).
  • Instance Recovery : Execute la sequence de recovery SMON : Redo Apply puis Undo Apply. Les marqueurs repassent au vert.
  • Commit (SCN++) : Incremente le SCN, simulant un commit. Le compteur SCN central s'actualise.
  • Query a SCN donne : Simule une Flashback Query (AS OF SCN), reconstruisant l'image des donnees a un SCN historique via l'Undo.
  • Simuler Lecture Coherente : Affiche deux sessions paralleles : Session A voit les donnees a son SCN de depart (reconstruites via Undo), Session B voit les modifications courantes. Demontre que Session A n'est pas affectee par les modifications de Session B.
  • UNDO_RETENTION slider : Definit la duree de conservation des Undo blocks (60 a 3600 secondes). Combine avec la duree de requete et la charge DML, determine le risque d'ORA-01555.
Indicateurs Cles
  • Compteur SCN : S'auto-incremente toutes les 2 secondes (simulation d'activite background). Chaque commit avance le SCN.
  • Marqueurs de flux : 5 etapes colorees (gris = en attente, bleu = en cours, vert = complete, rouge = erreur). Permettent de suivre visuellement la progression de la transaction.
  • Retention effective : Valeur calculee = UNDO_RETENTION * (1 - DML_load / 200). Si la duree de requete excede cette valeur, ORA-01555 se produit.
  • Log console ORA-01555 : Affiche la progression de la requete longue et le moment ou les Undo blocks sont ecrases.
Contexte HDS / CHU

La coherence de lecture est vitale pour la facturation T2A : une requete de facturation lancée a 23h doit produire le meme resultat qu'une requete identique lancée a 23h05, meme si des modifications sont en cours. Le SCN garantit cette propriete. L'Instance Recovery assure qu'aucune donnee n'est corrompue apres un crash, meme pendant un pic d'activite aux Urgences. L'ORA-01555 peut faire echouer les batchs de facturation nocturnes : un UNDO_RETENTION mal calibre (ou un tablespace UNDO sous-dimensionne) provoque l'echec de requetes de plusieurs heures.

Module 4 - Cloud Souverain & IA Controlee
Objectif de la Simulation

Ce module adresse les enjeux de souverainete des donnees de santé et d'usage de l'IA dans un contexte hospitalier reglemente (HDS, RGPD, ANS) :

  • TDE (Transparent Data Encryption) AES-256 : Chiffrement des donnees au repos (data-at-rest) au niveau tablespace. La cle maitresse est stockee dans un HSM (Hardware Security Module). En mode BYOK (Bring Your Own Key), le CHU controle ses propres cles, independamment d'Oracle.
  • Auto-Tuning (IA embarquee) : L'Autonomous Database integre un moteur d'optimisation local : creation automatique d'indexes, SQL Plan Management (SPM), collecte adaptive de statistiques. Aucune donnee ne quitte le perimetre OCI France.
  • IA embarquee vs IA generative externe : Distinction methodologique critique. L'IA embarquee (Auto-Tuning) s'execute localement sur l'instance, sans transfert de donnees. L'IA generative externe (LLM cloud) implique un transfert de donnees hors du perimetre HDS, soumis au RGPD Art. 44-49 (transferts internationaux).
  • Souverainete OCI France : Architecture en couches : HSM (cles) -> TDE (chiffrement) -> Autonomous DB (eu-france-1) -> Data CHU. Chaque couche est certifiee HDS par l'ANS.
Controles Disponibles
  • Chiffrer (TDE AES-256) : Simule le chiffrement des champs sensibles (NIR, nom, diagnostic). Affiche les cles hexadecimales simulees avec les metadonnees de chiffrement (Key ID, algorithme, mode).
  • Dechiffrer (Cle HSM) : Tente de dechiffrer avec la cle HSM. Si la cle a ete revoquee, l'operation echoue (ORA-28365), demontrant que sans la cle, les donnees sont irrecuperables.
  • Revoquer Cle (BYOK) : Simule la rotation de cle maitresse. Apres revocation, le dechiffrement est impossible jusqu'a la re-importation de la nouvelle cle. En production HDS, une periode de coexistence des cles est necessaire.
  • Lancer Auto-Tuning : Execute une simulation d'analyse de plan d'execution avec 3 recommandations : creation d'index, SQL Plan Baseline, et ajustement des statistiques. Affiche l'amelioration de performance estimee (x21).
Indicateurs Cles
  • Statut TDE : V$ENCRYPTED_TABLESPACES: ENCRYPTED confirme le chiffrement actif. Overhead CPU : ~3-5%.
  • Revoquer Cle : ORA-28365: wallet is not open confirme que la revocation rend les donnees inaccesibles. C'est le comportement attendu et desire.
  • Auto-Tuning : V$SQL_AUTO_INDEX: INDEX CREATED et SQL_PLAN_BASELINE: ACCEPTED confirment les optimisations appliquees automatiquement.
  • Tableau comparatif IA : Compare IA embarquee vs generative externe sur 5 criteres : localisation des donnees, conformite RGPD, certification HDS, controle humain, et cas d'usage.
Contexte HDS / CHU

Dans un contexte HDS, le chiffrement TDE est une mesure de securite obligatoire (Art. 32 RGPD : securite du traitement). Le mode BYOK garantit que le CHU conserve la souverainete sur ses cles de chiffrement, independamment du fournisseur cloud. La distinction entre IA embarquee et IA generative externe est methodologiquement critique : l'IA embarquee ne pose aucun probleme de transfert de donnees de santé (les donnees restent dans OCI France, region certifiee HDS). L'IA generative externe necessite une analyse d'impact RGPD (DPIA) et potentiellement des clauses contractuelles pour les transferts hors UE. L'approche Human-in-the-Loop garantit que toute recommandation d'IA (auto-tuning) est soumise a validation humaine avant application en production.
