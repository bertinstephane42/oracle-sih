💣 Code à ne JAMAIS écrire
Interdit — CHECK non-déterministe
CREATE TABLE prescriptions (
  date_creation DATE,
  CONSTRAINT ck_date CHECK(SYSDATE >= date_creation)  -- ERREUR
);
Solution — Trigger BEFORE
CREATE OR REPLACE TRIGGER trg_prescriptions_before
BEFORE INSERT OR UPDATE ON prescriptions
FOR EACH ROW
BEGIN
  IF :NEW.date_creation > SYSDATE THEN
    RAISE_APPLICATION_ERROR(-20001,
      'La date de création ne peut pas être future.');
  END IF;
END;
Interdit — COMMIT dans trigger
CREATE OR REPLACE TRIGGER trg_after_insert
AFTER INSERT ON prescriptions
BEGIN
  COMMIT;  -- ORA-04092
END;
Solution — Lever une exception
CREATE OR REPLACE TRIGGER trg_after_insert
AFTER INSERT ON prescriptions
BEGIN
  RAISE_APPLICATION_ERROR(-20002,
    'Insertion refusée : règle métier non respectée.');
END;
🏗️ 4 Règles d'Or de la Conception
1. Indexer les Clés Étrangères

Sans index sur une FK, toute suppression dans la table parent verrouille la table enfant entièrement (TM Lock).

-- À faire sur TOUTE FK :
CREATE INDEX idx_lots_med_id
  ON lots(med_id) TABLESPACE pharma_index;
2. Séparer DATA et INDEX

Tablespace distincts pour les données et les index = meilleure gestion I/O et maintenance.

CREATE TABLESPACE pharma_data
  DATAFILE '/u01/oradata/pharma_data.dbf'
  SIZE 10G AUTOEXTEND ON;

CREATE TABLESPACE pharma_index
  DATAFILE '/u01/oradata/pharma_index.dbf'
  SIZE 2G AUTOEXTEND ON;
Bonne pratique historique essentielle sur fichiers systèmes traditionnels. Sous ASM ou stockage SSD/NVMe, l'impact est moindre mais la séparation facilite toujours la maintenance (rebuild d'index sans contention I/O sur les données).
3. Préférer IDENTITY aux Séquences

Les colonnes IDENTITY (12c+) évitent la gestion manuelle des séquences et garantissent l'intégrité.

CREATE TABLE prescriptions (
  id NUMBER
    GENERATED BY DEFAULT AS IDENTITY,
  ...
);
4. Toujours VARCHAR2, Jamais CHAR

CHAR(10) alloue toujours 10 caractères, même pour un code de 3 lettres. VARCHAR2 n'alloue que le nécessaire.

-- MAUVAIS : CHAR(10) → 10 octets fixes
code_lot CHAR(10);

-- BON : VARCHAR2(10) → taille réelle
code_lot VARCHAR2(10);

⚡ Set-Based Mindset : Le Réflexe Qui Sauve

"Si c'est faisable en SQL pur, ne fais jamais de PL/SQL bouclant." — Chaque itération de boucle PL/SQL coûte un context switch (SQL ↔ PL/SQL) d'environ 5 μs. Multipliez par des millions de lignes, et le temps d'exécution explose.

🔥 Piège Fatal : Calcul sur Colonne Indexée
WHERE TO_CHAR(date_creation, 'YYYY-MM-DD') = '2026-01-15' ignore l'index !
-- MAUVAIS : fonction sur la colonne = FULL SCAN
SELECT * FROM prescriptions
WHERE TO_CHAR(date_creation, 'YYYY-MM-DD')
  = '2026-01-15';

-- BON : prédicat de plage = INDEX RANGE SCAN
SELECT * FROM prescriptions
WHERE date_creation >= DATE '2026-01-15'
  AND date_creation <  DATE '2026-01-16';
📦 BULK COLLECT / FORALL

Une boucle FETCH ligne à ligne provoque N context switches (SQL ↔ PL/SQL, ≈ 5 μs chacun). BULK COLLECT réduit à 1 aller-retour pour récupérer toutes les lignes, et FORALL à 1 aller-retour pour les mettre à jour. Sur 100 000 lignes, le gain est de plusieurs ordres de grandeur.

DECLARE
  TYPE t_ids IS TABLE OF prescriptions.id%TYPE;
  v_ids t_ids;
BEGIN
  -- Récupération massive en mémoire
  SELECT id BULK COLLECT INTO v_ids
  FROM prescriptions
  WHERE statut = 'EN_ATTENTE'
  FOR UPDATE;

  -- Mise à jour par lots (pas ligne à ligne)
  FORALL i IN 1..v_ids.COUNT
    UPDATE prescriptions
    SET statut = 'TRAITEE'
    WHERE id = v_ids(i);

  COMMIT;
END;
📊 Statistiques CBO : Toujours à Jour

L'Optimiseur Coût-Based (CBO) prend ses décisions sur la base des statistiques. Des stats obsolètes = un mauvais plan d'exécution.

BEGIN
  DBMS_STATS.GATHER_TABLE_STATS(
    ownname          => 'PHARMA',
    tabname          => 'PRESCRIPTIONS',
    estimate_percent => DBMS_STATS.AUTO_SAMPLE_SIZE,
    method_opt       => 'FOR ALL COLUMNS SIZE AUTO',
    degree           => DBMS_STATS.AUTO_DEGREE,
    cascade          => TRUE
  );
END;
🔍 Générateur de Requête V$SESSION

Cochez les colonnes que vous voulez surveiller. Le SQL de diagnostic est généré en temps réel. V$SESSION a 300+ colonnes ; un filtre expert n'en utilise que 15.

Requête générée :
SELECT s.sid,
       s.serial#,
       s.username,
       s.status,
       s.event,
       s.seconds_in_wait
FROM   v$session s
WHERE  s.type != 'BACKGROUND'
ORDER  BY s.seconds_in_wait DESC NULLS LAST;
🚨 ORA-00054 : Resource Busy

Se produit quand une commande DDL (DROP, TRUNCATE, ALTER) tente d'acquérir un verrou exclusif sur un objet qu'une transaction DML concurrente utilise déjà. Solution :

-- Trouver le bloqueur
SELECT sid, serial#, username, machine
FROM   v$session
WHERE  sid IN (
  SELECT blocking_session
  FROM   v$session
  WHERE  blocking_session IS NOT NULL
);

-- Kill (en dernier recours)
ALTER SYSTEM KILL SESSION '123,456';

-- Kill niveau OS (si SQL*Plus injoignable)
-- kill -9 <SPID>

💡 La Requête "Couteau Suisse" Complète
SELECT s.sid,
       s.serial#,
       s.username,
       s.status,
       s.event,
       s.seconds_in_wait,
       s.sql_id,
       s.blocking_session,
       s.machine,
       s.program,
       p.spid                         AS pid_os,
       q.sql_text
FROM   v$session s
       LEFT JOIN v$process p
         ON p.addr = s.paddr
       LEFT JOIN v$sql q
         ON q.sql_id = s.sql_id
        AND q.child_number = s.sql_child_number
WHERE  s.type != 'BACKGROUND'
AND    (   s.status = 'ACTIVE'
        OR s.seconds_in_wait > 0)
ORDER  BY s.seconds_in_wait DESC NULLS LAST;
🧠 La Philosophie du DBA Senior
🎯 Apprentissage Just-in-Time

Ne lisez pas le manuel Oracle en entier. Lancez-vous sur un cas concret, et cherchez la solution au moment où vous en avez besoin. La théorie sans pratique s'oublie en 48h. La pratique ciblée avec documentation ponctuelle s'ancre durablement.

"Je ne connais pas la syntaxe de MERGE. Je la cherche dans 5 minutes quand j'en aurai besoin."
📖 Le Dictionnaire de Données est ta Bible

Avant Google, tapez DICT et DICT_COLUMNS. Ces vues contiennent la liste de TOUTES les vues du dictionnaire et leurs colonnes. Une compétence qui différencie immédiatement un DBA senior d'un débutant. 

-- Trouver la vue qui contient 'TABLESPACE'
SELECT table_name, comments
FROM   dict
WHERE  table_name LIKE '%TABLESPACE%'
ORDER  BY table_name;
🔐 Principe du Moindre Privilège

Jamais de GRANT DBA à un utilisateur métier. Créez des rôles spécifiques avec les privilèges strictement nécessaires. C'est la règle #1 de la sécurité. 

CREATE USER med_pharma
  IDENTIFIED BY "Str0ng!Pass"
  DEFAULT TABLESPACE pharma_data
  QUOTA UNLIMITED ON pharma_data;

CREATE ROLE role_prescri;
GRANT SELECT, INSERT, UPDATE
  ON prescriptions TO role_prescri;
GRANT role_prescri TO med_pharma;

Pourquoi ces interdits ?
Principe fondamental

Oracle impose des contraintes de déterministe pour les contraintes CHECK : une fonction appelée dans un CHECK doit retourner la même valeur pour les mêmes données d'entrée, peu importe quand elle est exécutée. SYSDATE viole cette règle — sa valeur change à chaque milliseconde.

De même, les triggers font partie de la transaction appelante. Un COMMIT dans un trigger briserait l'atomicité de la transaction, d'où l'interdiction formelle (ORA-04092).

Pourquoi éviter SYS ?

Le schéma SYS contient le dictionnaire de données. Créer des objets métier dedans expose au risque de corruption lors des opérations de maintenance Oracle (upgrade, patch). Oracle ne supporte pas les objets utilisateur dans SYS.

Architecture — Les Fondations
Pourquoi indexer les FK ?

Quand vous supprimez une ligne dans une table parent, Oracle doit vérifier qu'aucune ligne enfant ne référence la ligne supprimée. Sans index sur la FK, cette vérification verrouille toute la table enfant (TM Lock). Avec index, seule la ligne concernée est verrouillée (Row Lock). Sur une table de millions de lignes, la différence est vitale.

Séparation DATA / INDEX

En plaçant les données et les index dans des tablespaces distincts (sur des disques différents si possible), vous réduisez la contention I/O et facilitez la maintenance (ex: rebuild d'index sans impact sur les données).

Set-Based Mindset
Context Switching

Chaque passage du moteur SQL au moteur PL/SQL coûte environ 5 microsecondes (context switch). Dans une boucle traitant 100 000 lignes, ce coût atteint 0,5 seconde uniquement en changements de contexte — sans compter le traitement lui-même. Un UPDATE massif SQL traite toutes les lignes en une seule opération sans context switch : le gain est exponentiel.

Fonction sur colonne indexée

Quand vous appliquez une fonction à une colonne indexée dans le WHERE (comme TO_CHAR(date)), l'optimiseur ne peut pas utiliser l'index car la valeur indexée brute ne correspond plus à la valeur recherchée après transformation. Utilisez toujours un prédicat de plage sur la colonne native.

Monitoring V$SESSION
Pourquoi ne pas tout sélectionner ?

V$SESSION contient plus de 300 colonnes. Les charger toutes alourdit inutilement le diagnostic. Les 15 colonnes vitales (SID, USERNAME, STATUS, EVENT, SECONDS_IN_WAIT, SQL_ID, BLOCKING_SESSION, etc.) suffisent dans 95% des cas pour identifier un problème. L'art du DBA est de savoir ce qu'il faut ignorer.

SECONDS_IN_WAIT > 300s

Une session qui attend plus de 300 secondes (5 minutes) sur un événement de type lock ou I/O est un signal d'alarme critique. Dans un SIH, cela peut signifier qu'une prescription est bloquée, avec un impact patient potentiel.

Mindset & Méthodologie
Apprentissage Just-in-Time

La méthode traditionnelle (lire un manuel de 1000 pages avant de pratiquer) est inefficace pour la mémoire de travail. L'approche Just-in-Time : face à un problème concret (ex: "comment partitionner cette table ?"), cherchez la solution immédiatement, appliquez-la, et les connaissances s'ancrent par l'action. Vous construisez votre expertise par accumulation de cas résolus.

Pourquoi DICT est magique

La vue DICT (et sa version colonnes DICT_COLUMNS) est le moteur de recherche interne d'Oracle. Au lieu de googler "Oracle view tablespace usage", tapez SELECT * FROM DICT WHERE table_name LIKE '%TABLESPACE%'. Vous obtenez instantanément toutes les vues disponibles, sans connexion internet, sans attente, sans bruit.

Les vues "magiques"

DICT : liste toutes les vues du dictionnaire. DICT_COLUMNS : liste les colonnes de ces vues. Les préfixes : USER_ (vos objets), ALL_ (objets accessibles), DBA_ (tous les objets), V$_ (vues dynamiques de performance). Maîtrisez ces préfixes et vous naviguez le dictionnaire comme personne.

Lexique technique — Manuel de Survie
Concepts Fondamentaux
  • Déterministe — Se dit d'une fonction qui retourne toujours la même valeur pour les mêmes arguments. Les contraintes CHECK exigent des fonctions déterministes. SYSDATE, USER, RANDOM sont non-déterministes.
  • Set-Based — Approche de programmation SQL qui traite les données par ensembles (sets) plutôt que ligne à ligne. Opposé à la pensée procédurale (boucle). "Parle en ensembles, pas en lignes."
  • Context Switching — Changement de contexte entre le moteur SQL et le moteur PL/SQL. Chaque switch coûte ~5μs. Une boucle PL/SQL sur 100k lignes paie 500ms uniquement en changements de contexte.
  • CBO (Cost-Based Optimizer) — Moteur d'optimisation d'Oracle qui choisit le plan d'exécution le moins coûteux pour une requête SQL, en se basant sur les statistiques des objets (DBMS_STATS).
  • IDENTITY (Colonne) — Colonne générée automatiquement (Oracle 12c+) similaire à AUTO_INCREMENT. GENERATED BY DEFAULT AS IDENTITY permet l'insertion manuelle si nécessaire.
Verrous et Concurrence
  • Row Lock — Verrouillage au niveau de la ligne (TX Lock). Permet à plusieurs transactions de modifier différentes lignes simultanément. C'est le verrouillage par défaut d'Oracle.
  • TM Lock (DML Lock) — Verrou de table (Table Management Lock). Protège la structure de la table contre les DDL concurrents. Un DELETE sur un parent sans index FK peut escalader en TM Lock.
  • BLOCKING_SESSION — Colonne de V$SESSION qui indique le SID de la session bloquante. NULL si la session n'est pas bloquée. Permet de remonter la chaîne de blocage.
  • ORA-00054 — Erreur "resource busy and acquire with NOWAIT specified". Se produit quand un DDL rencontre un verrou DML concurrent.
  • ORA-04092 — Erreur "cannot COMMIT in a trigger". Oracle interdit les COMMIT/ROLLBACK dans les triggers car ils brisent l'atomicité de la transaction.
Performance et Monitoring
  • Wait Event — Événement pour lequel une session Oracle attend. Classes : CPU (ON CPU), I/O (db file scattered read), Lock (enq: TX). La colonne SECONDS_IN_WAIT donne la durée.
  • SPID (System PID) — Identifiant de processus OS correspondant à une session Oracle. Jointure V$SESSION → V$PROCESS (colonne SPID). Utile pour kill -9 en dernier recours.
  • BULK COLLECT — Instruction PL/SQL qui récupère toutes les lignes d'une requête en une seule opération, évitant les context switches répétés.
  • FORALL — Instruction PL/SQL qui exécute un DML pour chaque élément d'une collection en un seul context switch. Complémentaire de BULK COLLECT.
  • DBMS_STATS — Package de collecte des statistiques pour le CBO. Des statistiques fraîches sont essentielles pour des plans d'exécution optimaux.
Dictionnaire et Outils
  • DICT / DICT_COLUMNS — Vues du dictionnaire listant toutes les vues et colonnes du dictionnaire Oracle. L'outil de recherche interne le plus puissant avant même Google.
  • RMAN (Recovery Manager) — Outil de sauvegarde et restauration intégré à Oracle. Seul outil supporté pour les backups cohérents. Permet les restaurations point-in-time.
