-- =============================================================================
-- GESTION DE L'ALLOCATION ET DU CHIFFREMENT DES TABLESPACES - PHARMA_DATA (PUI)
-- =============================================================================
-- Objectif : Garantir la continuité de service de la Pharmacie à Usage Intérieur
-- en évitant la saturation des tablespaces et en assurant le chiffrement des
-- données de santé au repos (HDS §3.2).
--
-- Contexte : Le tablespace PHARMA_DATA stocke l'ensemble des données de
-- prescription, de stock et de dispensation. Une saturation bloque toute
-- prescription ; un datafile non chiffré expose les données patients.
--
-- Étapes du script :
--   1. Allocation (ALTER ADD DATAFILE) avec MAXSIZE pour borner la croissance.
--   2. Monitoring (DBA_TABLESPACE_USAGE_METRICS, seuil d'alerte à 80 %).
--   3. Vue globale : inventaire, occupation, audit autoextend, temp, UNDO.
--   4. Contrôle du chiffrement (V$ENCRYPTION_WALLET — HDS §3.2).
--
-- Bonne pratique HDS : MAXSIZE impératif sur chaque datafile + alerte
-- de supervision à 80 % d'occupation. Chiffrement AES-256 obligatoire
-- via CREATE TABLESPACE ... ENCRYPTION USING 'AES256'.
-- =============================================================================
--
-- Créer un tablespace chiffré (TDE AES-256 — exigence HDS §3.2)
-- CREATE TABLESPACE pharma_data ENCRYPTION USING 'AES256'
--   DATAFILE 'pharma01.dbf' SIZE 1G AUTOEXTEND ON MAXSIZE 10G;

-- Ajouter un fichier de données avec taille maximale
ALTER TABLESPACE pharma_data
ADD DATAFILE 'pharma02.dbf' SIZE 1G
AUTOEXTEND ON MAXSIZE 10G;

-- Vérifier la taille allouée
SELECT tablespace_name, SUM(bytes)/1024/1024 AS size_mb
FROM dba_data_files
WHERE tablespace_name = 'PHARMA_DATA'
GROUP BY tablespace_name;

-- Vérifier l'espace libre
SELECT tablespace_name, SUM(bytes)/1024/1024 AS free_mb
FROM dba_free_space
WHERE tablespace_name = 'PHARMA_DATA'
GROUP BY tablespace_name;

-- Métriques d'usage
SELECT tablespace_name, used_space, tablespace_size, used_percent,
       CASE WHEN used_percent >= 80 THEN '⚠ ALERTE' ELSE 'OK' END AS seuil_hds
FROM dba_tablespace_usage_metrics
WHERE tablespace_name = 'PHARMA_DATA';

-- === VUE GLOBALE (après création) ===

-- 1. Lister TOUS les tablespaces avec état, type et chiffrement
SELECT tablespace_name, status, contents,
       extent_management, segment_space_management,
       logging, force_logging, encrypted, block_size, allocation_type
FROM dba_tablespaces
ORDER BY tablespace_name;

-- 2. Occupation globale (tous tablespaces confondus) avec seuil d'alerte
SELECT df.tablespace_name,
       ROUND(SUM(df.bytes)/1024/1024) AS total_mb,
       ROUND(SUM(fs.bytes)/1024/1024) AS free_mb,
       ROUND((1 - NVL(SUM(fs.bytes),0)/NULLIF(SUM(df.bytes),0)) * 100, 1) AS used_pct,
       CASE WHEN ROUND((1 - NVL(SUM(fs.bytes),0)/NULLIF(SUM(df.bytes),0))*100,1) >= 80
            THEN '⚠ ALERTE' ELSE 'OK' END AS seuil_hds,
       ROUND(SUM(maxbytes)/1024/1024) AS max_mb_agrege
FROM dba_data_files df
LEFT JOIN dba_free_space fs ON df.file_id = fs.file_id
GROUP BY df.tablespace_name
ORDER BY used_pct DESC;

-- 3. Audit des datafiles (autoextend, statut, croissance restante)
SELECT tablespace_name, file_name,
       ROUND(bytes/1024/1024) AS size_mb,
       autoextensible,
       ROUND(maxbytes/1024/1024) AS max_mb,
       ROUND((maxbytes - bytes)/1024/1024) AS remaining_growth_mb,
       status, online_status
FROM dba_data_files
ORDER BY tablespace_name, file_name;

-- 4. Tablespace temporaire : capacité totale et espace utilisé
SELECT tf.tablespace_name,
       ROUND(SUM(tf.bytes)/1024/1024) AS total_mb,
       ROUND(SUM(tsh.bytes_used)/1024/1024) AS used_mb,
       ROUND(SUM(tsh.bytes_free)/1024/1024) AS free_mb
FROM dba_temp_files tf
LEFT JOIN v$temp_space_header tsh ON tf.tablespace_name = tsh.tablespace_name
GROUP BY tf.tablespace_name;

-- 5. Tablespace UNDO : rétention, taille et prévention ORA-01555
SELECT ts.tablespace_name, ts.retention,
       ROUND(SUM(df.bytes)/1024/1024) AS size_mb,
       (SELECT ROUND(MAX(maxquerylen)) FROM v$undostat) AS plus_longue_requete_secondes
FROM dba_tablespaces ts
JOIN dba_data_files df ON ts.tablespace_name = df.tablespace_name
WHERE ts.contents = 'UNDO'
GROUP BY ts.tablespace_name, ts.retention;

-- 6. Statut du wallet TDE (chiffrement données au repos — HDS §3.2)
SELECT wrl_type, wrl_parameter, status, wallet_type
FROM v$encryption_wallet;

-- =============================================================================
-- GESTION DU CYCLE DE VIE DES STATISTIQUES - TABLE SORTIE_STOCK (PUI)
-- =============================================================================
-- Objectif : Maintenir la performance du CBO (Cost-Based Optimizer) pour les
-- requêtes critiques de la Pharmacie à Usage Intérieur, tout en garantissant
-- la stabilité des plans d'exécution en production (prévention des régressions).
--
-- Contexte : La table SORTIE_STOCK est soumise à de fortes variations de volume
-- et de distribution (pics d'activité, nouveaux médicaments). Des statistiques
-- obsolètes entraînent des scans complets (FULL TABLE SCAN) au lieu d'utiliser
-- les index, provoquant des ralentissements lors des tournées de soins.
--
-- Étapes du script :
--   1. COLLECTE (GATHER) : Mise à jour complète stats table + index (cascade)
--      avec histogramme automatique sur les colonnes skewées.
--   2. AUDIT : Vérification de la date de mise à jour et du compteur de lignes.
--   3. SÉCURISATION (LOCK) : Verrouillage des stats après validation pour
--      empêcher Oracle de les recalculer intempestivement.
--   4. SURVEILLANCE : Détection des tables 'STALE' et état des stats index.
--
-- Bonne pratique HDS : Planifier la collecte (étape 1) durant les creux
-- d'activité (ex: 04h00). N'appliquer le verrou (étape 3) qu'après une
-- période de recette confirmant la stabilité des performances.
-- =============================================================================
--
BEGIN
  DBMS_STATS.GATHER_TABLE_STATS(
    ownname    => 'PHARMA',
    tabname    => 'SORTIE_STOCK',
    cascade    => TRUE,
    method_opt => 'FOR ALL COLUMNS SIZE AUTO'
  );
END;

-- Vérifier la date des dernières statistiques
SELECT table_name, last_analyzed, num_rows
FROM dba_tables
WHERE owner = 'PHARMA' AND table_name = 'SORTIE_STOCK';

-- Verrouiller les statistiques après validation des plans (prévention régression HDS)
BEGIN DBMS_STATS.LOCK_TABLE_STATS(ownname => 'PHARMA', tabname => 'SORTIE_STOCK'); END;

-- Vérifier le verrouillage
SELECT table_name, stattype_locked FROM dba_tab_statistics
WHERE owner = 'PHARMA' AND table_name = 'SORTIE_STOCK';

-- Détecter les statistiques obsolètes
SELECT table_name, stale_stats, last_analyzed
FROM dba_tab_statistics
WHERE owner = 'PHARMA' AND stale_stats = 'YES';

-- Vérifier les statistiques des index associés
SELECT index_name, last_analyzed, stale_stats, sample_size
FROM dba_ind_statistics
WHERE table_owner = 'PHARMA' AND table_name = 'SORTIE_STOCK';

-- GESTION DU RECYCLE BIN - RÉCUPÉRATION APRÈS SUPPRESSION ACCIDENTELLE (PUI)

-- Objectif : Permettre la récupération rapide d'une table supprimée
-- accidentellement en production, tout en maîtrisant l'impact sur l'espace
-- disque et la traçabilité HDS.
--
-- Contexte : Une suppression accidentelle de table (DROP TABLE) dans une PUI
-- peut interrompre des processus critiques de dispensation. Le Recycle Bin
-- Oracle offre un filet de sécurité immédiat via FLASHBACK.
--
-- Étapes du script :
--   1. AUDIT : Visualisation des objets supprimés (USER_RECYCLEBIN / DBA).
--   2. RÉCUPÉRATION : FLASHBACK TABLE, avec ou sans renommage.
--   3. PURGE : Libération d'espace maîtrisée et sélective.
--   4. DÉFINITIF : DROP TABLE ... PURGE (contournement du Recycle Bin).
--
-- Bonne pratique HDS : Toute PURGE définitive supprime la traçabilité —
-- préférer DROP TABLE simple + purge différée après audit. La vue
-- DBA_RECYCLEBIN permet une supervision multi-utilisateurs.
-- =============================================================================
--
-- Voir les objets dans le Recycle Bin
SELECT object_name, original_name, type, droptime
FROM user_recyclebin;

-- Vue DBA : objets supprimés de tous les utilisateurs
SELECT owner, object_name, original_name, droptime, space
FROM dba_recyclebin
ORDER BY owner, droptime DESC;

-- Restaurer une table
FLASHBACK TABLE stock_med TO BEFORE DROP;

-- Restaurer avec un nouveau nom (si l'original existe encore)
FLASHBACK TABLE stock_med TO BEFORE DROP RENAME TO stock_med_recovered;

-- Purge sélective d'une table
PURGE TABLE stock_med;

-- Vider le Recycle Bin
PURGE RECYCLEBIN;

-- Suppression simple (passe par le Recycle Bin — recommandé)
DROP TABLE stock_med;

-- DROP sans passer par le Recycle Bin
DROP TABLE stock_med PURGE;

-- =============================================================================
-- GESTION DU COMPROMIS LECTURE / ÉCRITURE - INDEXATION DE SORTIE_STOCK (PUI)
-- =============================================================================
-- Objectif : Optimiser les performances des requêtes de lecture (dispensation,
-- inventaire) sans dégrader les écritures transactionnelles (entrées-sorties
-- de stock) en maîtrisant le nombre, l'état et la fragmentation des index.
--
-- Contexte : La table SORTIE_STOCK est interrogée en temps réel par les
-- préparateurs (lecture) et mise à jour à chaque mouvement de stock (écriture).
-- Un excès d'index ralentit les écritures ; un défaut pénalise les lectures.
--
-- Étapes du script :
--   1. CRÉATION : CREATE INDEX sur les colonnes de filtrage critiques.
--   2. SUIVI : MONITORING USAGE + V$OBJECT_USAGE (détection des inutilisés).
--   3. FRAGMENTATION : ANALYZE INDEX + INDEX_STATS (seuil > 30 % → REBUILD).
--   4. SÉCURISATION : ALTER INDEX INVISIBLE avant suppression (safety net).
--   5. SUPPRESSION : DROP INDEX après confirmation d'absence d'impact.
--
-- Bonne pratique HDS : Rendre l'index INVISIBLE avant suppression pour
-- observer l'impact en production sans perte de données. Un index
-- inutilisé dégrade les DML sans bénéfice (risque de performance évitable).
-- =============================================================================
--
-- Créer un index
CREATE INDEX idx_sortie_date
ON sortie_stock(date_sortie);

-- Lister les index d'une table
SELECT index_name, uniqueness, status, visibility
FROM user_indexes
WHERE table_name = 'SORTIE_STOCK';

-- Activer le suivi d'utilisation (détection index inutilisés)
ALTER INDEX idx_sortie_date MONITORING USAGE;
SELECT index_name, table_name, monitoring, used, start_monitoring, end_monitoring
FROM v$object_usage WHERE index_name = 'IDX_SORTIE_DATE';

-- Vérifier l'état physique d'un index
ANALYZE INDEX idx_sortie_date VALIDATE STRUCTURE;
SELECT name, height, lf_rows, del_lf_rows,
  ROUND(del_lf_rows / NULLIF(lf_rows, 0) * 100, 1) AS pct_frag
FROM index_stats;

-- Reconstruire un index fragmenté (> 30 % de lignes supprimées)
ALTER INDEX idx_sortie_date REBUILD ONLINE;

-- Rendre invisible avant suppression (safety net SIH)
ALTER INDEX idx_sortie_date INVISIBLE;

-- Vérifier que l'index n'est plus utilisé avant suppression
SELECT index_name, used, start_monitoring, end_monitoring
FROM v$object_usage
WHERE index_name = 'IDX_SORTIE_DATE';

-- Supprimer un index inutilisé (vérifier v$object_usage au préalable)
DROP INDEX idx_sortie_date;

-- =============================================================================
-- PRÉVENTION ET RÉSOLUTION DES INTERBLOCAGES - TRANSACTIONS PUI (ORA-00060)
-- =============================================================================
-- Objectif : Détecter et résoudre les situations d'interblocage (deadlock)
-- entre transactions concurrentes sur les tables critiques de la PUI
-- (STOCK_MED, LOT), afin de garantir l'intégrité des données et la continuité
-- des traitements de dispensation.
--
-- Contexte : Plusieurs sessions mettent à jour simultanément STOCK_MED et LOT
-- (dispensation, réapprovisionnement, inventaire). Un ordre de verrouillage
-- différent entre transactions provoque un ORA-00060, entraînant le rollback
-- d'une transaction et une perte de travail utilisateur.
--
-- Étapes du script :
--   1. DÉTECTION : Sessions bloquantes (V$SESSION.blocking_session).
--   2. ANALYSE : Verrous TM/TX (V$LOCK) et objets verrouillés (V$LOCKED_OBJECT).
--   3. HIÉRARCHIE : Chaîne de blocage complète (CONNECT BY PRIOR).
--   4. DIAGNOSTIC : SQL de la session bloquante (V$SQL) + trace ORA-00060.
--   5. RÉSOLUTION : Génération de la commande KILL SESSION.
--
-- Bonne pratique HDS : Toutes les transactions doivent verrouiller dans le
-- même ordre (règle d'or). Privilégier des transactions courtes pour réduire
-- la fenêtre de conflit. Analyser systématiquement le fichier de trace
-- ORA-00060 pour identifier la cause racine.
-- =============================================================================
--
-- Identifier une session bloquante
SELECT blocking_session, sid, serial#, event
FROM v$session
WHERE blocking_session IS NOT NULL;

-- Bonne pratique : ordre de verrouillage
-- identique dans TOUTES les transactions

-- Identifier l'objet verrouillé
SELECT ao.object_name, lo.session_id, lo.oracle_username
FROM v$locked_object lo
JOIN dba_objects ao ON lo.object_id = ao.object_id;

-- Analyse des verrous TM/TX (plus universel)
SELECT sid, type, lmode, request, id1, id2
FROM v$lock
WHERE type IN ('TM', 'TX') AND lmode > 0;

-- Chaîne de blocage hiérarchique
SELECT sid, blocking_session, LEVEL AS niveau
FROM v$session
WHERE blocking_session IS NOT NULL
CONNECT BY PRIOR sid = blocking_session
START WITH blocking_session IS NULL;

-- Localiser le fichier de trace ORA-00060 (interblocage)
SELECT value AS background_dump_dest
FROM v$parameter
WHERE name = 'background_dump_dest';

-- Texte SQL de la session bloquante
SELECT s.sid, s.serial#, s.sql_id, q.sql_text
FROM v$session s
LEFT JOIN v$sql q ON s.sql_id = q.sql_id
WHERE s.blocking_session IS NOT NULL;

-- Générer la commande KILL pour les sessions bloquantes
SELECT 'ALTER SYSTEM KILL SESSION ''' || sid || ',' || serial# || ''' IMMEDIATE;' AS kill_cmd
FROM v$session
WHERE blocking_session IS NOT NULL;

Tablespace — Gestion de l'espace
Objectif pédagogique

Comprendre l'impact du mode d'allocation (AUTOEXTEND UNLIMITED vs MAXSIZE + ALERTE) sur la continuité d'une PUI. Un tablespace saturé bloque toute prescription.

Règle métier SIH
  • MAXSIZE : Toujours définir une taille maximale pour éviter la saturation disque.
  • Alerte à 80% : Mettre en place une supervision remontant une alerte avant saturation.
  • Datafile supplémentaire : Ajouter un datafile dès 80% d'occupation pour garantir la continuité.
Statistiques CBO — Optimizer
Objectif pédagogique

Le Cost-Based Optimizer (CBO) utilise les statistiques des tables et index pour choisir le plan d'exécution optimal. Des statistiques obsolètes conduisent à des choix sous-optimaux et des temps de réponse dégradés.

Bonnes pratiques
  • Fréquence : Lancer DBMS_STATS quotidiennement sur les tables critiques PUI.
  • Cascade : Utiliser cascade => TRUE pour mettre à jour les index simultanément.
  • Verrouillage : Verrouiller les statistiques après validation des plans (DBMS_STATS.LOCK_TABLE_STATS). Déverrouiller uniquement en fenêtre de maintenance.
Recycle Bin — FLASHBACK vs PURGE
Objectif pédagogique

Le Recycle Bin Oracle stocke les objets supprimés (DROP TABLE) et permet leur récupération via FLASHBACK TABLE ... TO BEFORE DROP. La purge définitive (PURGE RECYCLEBIN) libère l'espace mais interdit tout retour en arrière.

Bonnes pratiques
  • FLASHBACK prioritaire : Toujours tenter un flashback avant une restauration depuis une sauvegarde.
  • Purge maîtrisée : Ne purger que lorsque l'espace est critique et que les objets sont validés comme non nécessaires.
  • Traçabilité HDS : La PURGE contourne le Recycle Bin et supprime toute traçabilité — obligatoire en HDS. Préférer DROP TABLE simple + purge différée après audit.
  • Script d'audit : Vérifier régulièrement le contenu du Recycle Bin via USER_RECYCLEBIN.
Indexation — Compromis lecture/écriture
Objectif pédagogique

Les index accélèrent les lectures (SELECT) mais ralentissent les écritures (INSERT/UPDATE/DELETE) car chaque index doit être maintenu. Trouver le bon équilibre est crucial pour une application PUI aux deux types d'accès.

Règle métier
  • Ratio optimal : 3-5 index par table OLTP pour un bon équilibre.
  • Index composites : Préférer un index composite à plusieurs index mono-colonne.
  • Monitoring : Utiliser INDEX_STATS pour vérifier l'utilisation réelle des index.
Deadlocks — ORA-00060
Objectif pédagogique

Un deadlock (interblocage) survient lorsque deux transactions attendent mutuellement un verrou détenu par l'autre. Oracle détecte la situation et rollbacke l'une des transactions avec une erreur ORA-00060.

Prévention
  • Ordre de verrouillage : Toutes les transactions doivent verrouiller les ressources dans le même ordre.
  • Transactions courtes : Limiter la durée des transactions pour réduire la fenêtre de conflit.
  • Trace ORA-00060 : Analyser le fichier de trace généré dans background_dump_dest.
Dashboard — Niveau de Risque SIH
Interprétation

Le score SIH global (0-100) est la somme des points gagnés ou perdus sur chaque pilier. Un score < 40 est critique, 40-70 est élevé, > 70 est maîtrisé. Chaque action de l'utilisateur impacte le score en temps réel.

Pondération
  • Tablespace : 20 pts max
  • Statistiques CBO : 20 pts max
  • Recycle Bin : 15 pts max
  • Indexation : 20 pts max
  • Deadlocks : 25 pts max
Lexique technique — Best Practices SIH
Gestion de l'espace
  • Tablespace — Conteneur logique de stockage Oracle. Une PUI doit avoir des tablespaces dédiés par type de données.
  • Autoextend — Extension automatique d'un datafile. Sans MAXSIZE, le disque peut saturer (→ PUI bloquée).
Optimiseur et Statistiques
  • CBO — Cost-Based Optimizer. Moteur de plan d'exécution Oracle basé sur les statistiques.
  • DBMS_STATS — Package Oracle de collecte des statistiques. Essentiel pour des plans optimaux.
Recycle Bin et Récupération
  • Recycle Bin — Espace de stockage des objets supprimés (DROP). Permet le FLASHBACK réversible.
  • Flashback — Récupération d'une table supprimée via FLASHBACK TABLE ... TO BEFORE DROP.
Verrous et Deadlocks
  • Deadlock — Interblocage entre 2 transactions. Oracle le détecte et rollbacke l'une d'elles (ORA-00060).
  • ORA-00060 — Code d'erreur Oracle signalant un deadlock. Trace générée dans le fichier de log.
  • Lock — Verrou placé sur une ligne ou une table pendant une transaction.
  • Row Exclusive — Mode de verrouillage ligne le plus courant (RX). Permet les lectures concurrentes.
