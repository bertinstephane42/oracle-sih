Requetes & Logique Metier
Objectif

Cet onglet simule un environnement de developpement Oracle pour la gestion des stocks PUI. Il demonstre trois aspects du langage Oracle :

  • Requete SQL complexe : Jointures multiples entre STOCK_MED, LOT_SIMU, SERVICE_SIMU avec filtres sur les dates de peremption. Simule l extraction des stocks critiques necessitant un reapprovisionnement urgent.
  • PL/SQL BULK COLLECT + FORALL : Bloc procedural utilisant les collections Oracle pour mettre a jour massivement les niveaux de stock. Le BULK COLLECT INTO charge les resultats en memoire (PGA), puis le FORALL execute les updates en batch, reduisant les context switches entre le moteur SQL et PL/SQL.
  • TRIGGER trg_check_peremption : Trigger BEFORE INSERT OR UPDATE sur SORTIE_STOCK qui verifie que la date de peremption du lot n est pas depassee. Regle metier critique : empeche la dispensation de medicaments expires aux services cliniques.
Controles
  • Snippet Selector : Choisit le type de requete a charger dans l editeur. Chaque snippet est pre-ecrit avec syntaxe Oracle valide.
  • Editeur : Zone de saisie avec coloration syntaxique basique (keywords, strings, comments). Le texte est modifiable avant execution.
  • Executer : Lance la machine d etats (PARSING -> EXECUTING -> FETCHING -> COMPLETED) avec delais artificiels simulant la latence disque/reseau.
  • Reset : Recharge le snippet d origine dans l editeur.
Indicateurs
  • Machine d etats : 5 etapes colorees (gris=inactif, bleu=en cours, vert=termine, rouge=erreur). Chaque etape a un delai simule : PARSING=800ms, EXECUTING=1500ms, FETCHING=600ms.
  • Console de sortie : Logs structures avec timestamps et niveaux. Affiche les resultats du parsing, les erreurs ORA-xxxxx simulees, et les statistiques d execution.
Contexte HDS / PUI

Dans une PUI, les requetes de stock doivent garantir l integrite des donnees : un medicament expire ne doit jamais etre dispense (d ou le trigger). Les batchs nocturnes de reapprovisionnement utilisent BULK COLLECT pour minimiser l impact sur la production. Toutes les donnees affichees sont fictives (MED_TEST_XXX, LOT_SIMU_XXX). Aucune donnee patient reelle n est utilisee.

Resultats & Analyse de Performance
Objectif

Cet onglet analyse les performances des requetes executees via le simulateur Cost Based Optimizer (CBO) d Oracle :

  • Explain Plan : Arbre d operations simule montrant comment Oracle accede aux donnees. Les operations incluent TABLE ACCESS FULL, INDEX UNIQUE SCAN, HASH JOIN, SORT GROUP BY, NESTED LOOPS.
  • Metriques CBO : Cout total, cardinalite estimee, nombre de lignes, temps d execution simule.
  • Alertes d indexation : Detection automatique des TABLE ACCESS FULL sur des tables volumineuses, avec recommandation de creation d index.
Controles
  • Metriques : 4 cartes KPI mises a jour automatiquement apres chaque execution (Temps ms, Logical Reads, Physical Writes, Cout CBO).
  • Data Grid : Tableau de resultats peuple dynamiquement avec les donnees fictives retournees par le simulateur. Les colonnes dependent de la requete executee.
  • Explain Plan : Arbre visuel avec indentation, operations colorees, cout et cardinalite par noeud.
  • Alertes CBO : Apparaissent automatiquement si un Full Table Scan est detecte sur une grande table.
Indicateurs Cles
  • Cout CBO : Unite arbitraire du cout d execution. Un cout > 500 indique une requete potentiellement sous-optimale.
  • Logical Reads : Nombre de lectures logiques Oracle (consistent gets + db block gets) réalisées via le Buffer Cache. Valeur elevee = nombreuses lectures memoire.
  • Physical Writes : Nombre de blocs ecrits sur disque. Non-zero pour les operations DML (UPDATE/INSERT).
  • Ratio LR/PW : Si Physical Writes > 10% de Logical Reads, verifier la taille du Buffer Cache.
Contexte HDS / PUI

Un Full Table Scan sur STOCK_MED peut fortement dégrader les performances et augmenter les temps de réponse lors des opérations critiques de dispensation. Dans une PUI hospitaliere, cela impacte directement la disponibilite des medicaments aux services critiques (Urgences, Reanimation). L optimisation des plans d execution via l indexation appropriee est un enjeu de continuite de soin.

Maintenance & Gouvernance
Objectif

Cet onglet simule les operations DBA et de gouvernance des donnees dans un contexte HDS :

  • Tablespaces : Visualisation de l utilisation de l espace disque pour chaque tablespace. Alertes automatique a 80% (warning) et 90% (critique).
  • Gestion RBAC : Creation d utilisateurs avec attribution selective de roles. Le principe de Least Privilege est applique : le role DBA_PUI declenche une alerte de securite.
  • Audit HDS : Journal chiffre de toutes les actions realisees dans le Workbench. Chaque entree inclut timestamp, utilisateur, action, requete, statut. Conforme aux exigences HDS et ISO 27001.
  • Purge Anonymisee : Simulation de DBMS_REDACT sur les donnees anciennes (> seuil configurable). Conforme au RGPD Art. 17 (droit a l effacement) et Art. 25 (privacy by design).
Controles
  • Simuler Charge TS : Augmente l utilisation des tablespaces pour demontrer les seuils d alerte.
  • Creer Utilisateur : Formulaire avec username et selection de roles. Validation des regles de securite.
  • Seuil purge : Slider de 30 a 365 jours pour definir l anciennete des donnees a purger.
  • Exporter Audit : Simulation d export CSV du journal d audit.
Indicateurs
  • % utilisation TS : Barre colorisee (vert < 80%, amber 80-90%, rouge > 90%).
  • Count audit : Nombre d entrees dans le journal. Croit avec chaque action du Workbench.
  • Records anonymises : Nombre de lignes masquee par la purge.
Contexte HDS / PUI

La gouvernance des donnees de sante exige une tracabilite totale (audit), un controle d acces strict (RBAC), et un cycle de vie des donnees conforme au RGPD (purge). Les tablespaces doivent etre monitors en continu : un tablespace saturé peut empêcher certaines opérations critiques Oracle (INSERT, SORT, UNDO, TEMP) selon le tablespace concerné. La purge anonymisee garantit que les donnees de test/simulation ne persistent pas indefiniment, respectant le principe de minimisation (RGPD Art. 5).

Architecture Oracle et Multitenant
  • Conteneur de Base de Donnees (CDB) — Base de donnees racine chapeautant l'ensemble des bases de donnees placables dans l'architecture multitenant Oracle 12c et suivantes.
  • Base de Donnees Placable (PDB) — Base de donnees portable et isolee au sein d'un conteneur, apparaissant aux utilisateurs et aux applications comme une base de donnees classique.
  • Dictionnaire Partage — Ensemble de metadonnees de definition (structures de schemas, objets, privileges) mutualisees entre toutes les bases placables d'un meme conteneur.
  • Cache de Lignes du Dictionnaire — Structure memoire logeant dans la memoire partagee les lignes du dictionnaire, protegee par un verrou d'exclusion mutuelle (latch).
  • Masquage de Donnees — Technique d'obfuscation des donnees sensibles (numero d'inscription au repertoire, identite des patients) lors du clonage d'une base, conforme au Reglement General sur la Protection des Donnees.
Memoire Globale (SGA / PGA)
  • Memoire Globale de Systeme (SGA) — Zone memoire partagee par l'ensemble des processus Oracle, abritant le cache de tampons, le reservoir partage, le tampon de journalisation et les structures connexes.
  • Cache de Tampons — Portion de la memoire globale contenant les blocs de donnees lus depuis les fichiers disques, ordonnancee par l'algorithme du moindre recent usage.
  • Memoire Globale de Programme (PGA) — Zone memoire privee allouee a chaque session d'utilisation pour les operations de tri, de regroupement et de jointure.
  • Debordement Temporaire — Basculement des operations memoire vers les espaces temporaires sur disque lorsque la memoire globale de programme est insuffisante, entrainant une degradation sensible des temps d'execution.
  • Compteur d'Acces (Touch Count) — Indicateur du nombre de consultations d'un bloc dans le cache de tampons, determinant son placement dans la chaine du moindre recent usage.
  • Verrou d'Exclusion Mutuelle (Latch) — Mecanisme de verrouillage leger protegeant l'acces concurrent aux structures partagees de la memoire globale.
Transactions et Coherence
  • Numero de Changement Systeme (SCN) — Compteur sequentiel global horodatant chaque transaction et garantissant l'ordre causal des modifications dans l'ensemble de la base.
  • Journal de Retablissement (Redo) — Enregistrement sequentiel de toutes les modifications appliquees a la base, utilise pour la reprise apres incident par application differee.
  • Segment d'Annulation (Undo) — Blocs conservant l'image anterieure des donnees modifiees, necessaires a la lecture coherente et au retour en arriere des transactions.
  • Reprise apres Incident (Instance Recovery) — Processus automatique de retablissement de la base apres un arret brutal, combinant l'application du journal de retablissement (avance rapide) et l'application des segments d'annulation (retour arriere).
  • Coherence de Lecture — Mecanisme garantissant qu'une requete percoit un instantane coherent des donnees a un instant de reference, en utilisant les blocs d'annulation.
  • Erreur d'Instantané Trop Ancien (ORA-01555) — Erreur survenant lorsque les blocs d'annulation necessaires a la lecture coherente d'une longue requete ont ete ecrases par des transactions plus recentes avant la fin de la consultation.
Optimiseur et Plans d'Execution
  • Optimiseur Base sur les Couts (CBO) — Moteur d'optimisation d'Oracle analysant les statistiques des objets et les couts d'acces pour determiner le plan d'execution le plus efficace pour chaque requete.
  • Plan d'Execution — Arbre d'operations (acces tables, jointures, tris) genere par l'optimiseur pour executer une requete SQL, visible via la commande EXPLAIN PLAN.
  • Parcours Integral de Table (Full Table Scan) — Methode d'acces lisant l'integralite des blocs d'une table, utilisee en l'absence d'index adapte ou lorsque la table est de petite taille.
  • Parcours Unique par Index (Index Unique Scan) — Methode d'acces retournant au plus une ligne via un index unique, typique des recherches par cle primaire.
  • Jointure par Hash (HASH JOIN) / Boucles Imbriquees (NESTED LOOPS) — Algorithmes de jointure : le premier convient aux grands volumes sans index, le second aux petites tables indexees.
PL/SQL et Procedural
  • Chargement Massif en Collection (BULK COLLECT) — Directive PL/SQL chargeant un ensemble de lignes en une seule operation dans une collection memoire (PGA), reduisant les allees-retours entre les moteurs SQL et PL/SQL.
  • Mise a Jour Massive (FORALL) — Instruction PL/SQL executant une operation DML sur toutes les lignes d'une collection sans commutation de contexte individuelle, accelerant les traitements par lots.
  • Commutation de Contexte (Context Switch) — Basculement entre le moteur SQL et le moteur PL/SQL, dont le cout cumule peut etre significatif dans les boucles iteratives.
  • Declencheur (Trigger) — Bloc PL/SQL stocke s'executant automatiquement sur un evenement de modification (INSERT, UPDATE, DELETE) d'une table, utilise pour les regles metier et les contraintes complexes.
  • Collection PL/SQL — Structure de donnees programmee (table imbriquee, VARRAY, table associative) permettant de manipuler des ensembles de lignes en memoire.
Securite et Gouvernance HDS
  • Chiffrement Transparent (TDE) — Mecanisme de chiffrement des donnees au repos au sein des fichiers et des espaces de tables, sans aucune modification des applications clientes, utilisant l'algorithme AES a clef de 256 bits.
  • Moindre Privilege (Least Privilege) — Principe de securite selon lequel un utilisateur ne dispose que des droits strictement necessaires a ses fonctions, reduisant la surface d'attaque et les risques d'erreur.
  • Masquage Dynamique (DBMS_REDACT) — Paquetage Oracle permettant le masquage a la volee des donnees sensibles dans les resultats de requetes, sans alterer les donnees stockees.
  • Journal d'Activite (Audit HDS) — Enregistrement chiffre et horodate de l'ensemble des actions effectuees sur le systeme, garantissant la tracabilite requise par la certification Hebergement de Donnees de Sante.
